------------------------------------------------------------
PROJECT BACKGROUND
------------------------------------------------------------

An enterprise application generates customer letters in PDF format.

Every generated letter contains three kinds of content:

1. Static Content
   - Common text that is identical for every customer.

2. Dynamic Content
   - Customer-specific values retrieved from a database.
   - Examples:
     - Member Name
     - Member ID
     - Address
     - Provider Name
     - Case Number
     - Dates
     - Phone Numbers
     - Other customer-specific information

3. Conditional Content
   - Certain paragraphs, sentences, bullet points, or even individual words appear only when specific business rules are satisfied.

------------------------------------------------------------
TEMPLATE STRUCTURE
------------------------------------------------------------

Unfortunately, I do NOT have the original Word (.docx) template.

I only have screenshots of the template.

From the screenshots I observed the following structure:

Display Rule:
A&G_Process = SHP

<Paragraph(s)>

Display Rule:
A&G_Process = CC

<Paragraph(s)>

Display Rule:
A&G_Process = Medicaid

<Paragraph(s)>

Display Rule:
A&G_Process = QHP or EP

<Paragraph(s)>

There are approximately:

- 10 to 15 conditional sections
- Each section has 1 to 3 business conditions
- Dynamic placeholders appear throughout the document.

Examples of placeholders:

Case.RelatedMember.Name

Case.RelatedMember.MemberID

Case.ProviderName

Case.DateTimeReceived

Case.CaseNumber

etc.

Some paragraphs also contain inline conditions.

Example:

If A&G_Process = SHP
show Sentence A

If A&G_Process = CC
show Sentence B

If A&G_Process = Medicaid
show Sentence C

So validation is required not only at paragraph level, but also at sentence level and sometimes even word level.

------------------------------------------------------------
GENERATED LETTER
------------------------------------------------------------

The generated output is always a PDF.

Dynamic values may vary in length.

Example:

Hello Mr. John Smith.

may become

Hello Mr.
Jonathan Christopher Alexander Smith.

or addresses may wrap to multiple lines.

Therefore formatting differences like:

- Line wrapping
- Indentation
- Additional spaces
- Different line breaks

must NOT cause validation failures.

Only the actual content matters.

------------------------------------------------------------
DATABASE
------------------------------------------------------------

The database contains all dynamic values.

Examples:

Member Name

Member ID

Provider Name

Case Number

A&G_Process

Phone Number

Date

Address

etc.

These values determine:

- Which paragraphs should appear.
- Which paragraphs should NOT appear.
- Which sentences should appear.
- Which dynamic values should appear.

------------------------------------------------------------
VALIDATION REQUIREMENTS
------------------------------------------------------------

The validator should automatically verify:

✓ Dynamic values exactly match the database.

✓ Static text matches the template.

✓ Expected conditional paragraphs exist.

✓ Unexpected conditional paragraphs do NOT exist.

✓ Inline conditional sentences are correct.

✓ Conditional words are correct.

✓ Ignore formatting differences caused by long dynamic values.

Finally generate a validation report showing all mismatches.
